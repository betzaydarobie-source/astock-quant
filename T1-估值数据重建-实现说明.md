# T1 估值历史数据重建 —— 实现说明

负责人：factor-engineer
状态：完成

## 一句话

把 PE / PB 因子从「腾讯实时快照（历史 95% 空值）」改成「用 4 年日线股价 + 重建的
季度财报自己算」，覆盖率 **PE 5% → 96.4%、PB 5% → 100%**，并新增股息率、毛利率
两个价值/质量因子。全程严守防 look-ahead（按财报披露日对齐，不是报告期末日）。

## 问题（改造前）

`astock_source.get_financials` 把腾讯实时 PE/PB 快照只挂到「最新一期」财报记录上，
所有历史报告期的 pe/pb 字段都是 None。后果：

- PE/PB 因子在历史交易日 95% 是 NaN —— 价值选股的「便宜度」地基塌了。
- 财务数据每次都 live 拉网络，无缓存，回测不可复现。

## 方案

### 1. 财报历史重建 —— `astock_quant/data/fundamentals.py`（新建）

数据源：
- **财报指标**：akshare `stock_financial_abstract_ths`（同花顺源）。实测 HS300 每只票
  都有 2021Q1 至今约 21 期，含 EPS / 每股净资产(BVPS) / ROE / 营收 / 净利 /
  毛利率 / 净利率 / 资产负债率。
- **分红数据**：akshare `stock_fhps_detail_em`（东财），含「业绩披露日期」「现金分红
  比例（每10股派X元）」。

产物：每只票一份 `data_cache/{code}-fundamentals.csv`（独立命名空间，不污染现有
prices/moneyflow 缓存）。回测直接读缓存，可复现。

已为 stage1 30 只票构建缓存：30/30 成功，共 2585 期财报记录。
扩 HS300 用 `python -m astock_quant.scripts.build_fundamentals_cache --stage4`。

### 2. ★防 look-ahead —— 用「财报披露日」而非「报告期末日」对齐★

**核心陷阱**：报告期末日 ≠ 财报可见日。2023 年报报告期末是 2023-12-31，但实际
2024 年 4 月才披露。站在 2024-02-15 看，最新可用财报还是 2023 三季报。原
`align_financials_to_panel` 按报告期末日 ffill —— 会在 1~3 月用到「当时还没公布」
的财报，是典型未来函数（原代码 docstring 里也标注了这个 Stage 1 已知 TODO）。

**为什么不用 akshare 的「公告日期」**：实测 akshare 的 EM 系列接口
（stock_lrb_em / stock_yjbb_em）的 `date=` 历史快照是假的 —— 传 date='20240331'
返回的「公告日期」其实是该股最新一期财报的日期（2026-04），不是 2024Q1 那期的
真实披露日。这些接口拿不到逐期真实披露日。

**对策 —— 法定披露截止日做「保守可见日」**：A股 监管对定期报告披露有硬性截止日：
| 报告期 | 法定披露截止日 |
|---|---|
| 一季报 (Q1) | 当年 4/30 |
| 半年报 (Q2) | 当年 8/31 |
| 三季报 (Q3) | 当年 10/31 |
| 年报 (Q4) | 次年 4/30 |

`statutory_publish_date()` 把报告期映射到法定截止日，作为 `FinancialMetrics.publish_date`。

**为什么这样不造成未来函数**：绝大多数公司**提前**于法定截止日披露，用法定截止日
是「偏晚」的估计 —— 宁可晚几周才用上财报，绝不会早于真实披露日用它。即这是
**保守**口径，方向上只牺牲一点点信息时效，不泄漏未来。这是 A股 量化研究处理
「无逐期披露日」时的标准做法。

分红的可见日用真实数据：`stock_fhps_detail_em` 的「业绩披露日期」是可信的逐期真实
披露日（已实测）—— 但本实现把分红金额按报告期挂到对应财报记录上，对齐时复用该
财报记录的 `publish_date`（保守，分红预案通常与年报同期或稍晚公告）。

下游对齐：新增 `factors/fundamental.align_by_publish_date()` —— 按 `publish_date`
排序后 ffill。某交易日 T 只用 `publish_date <= T` 的财报。T1 所有估值/质量因子
都走这条对齐路径。旧的 `align_financials_to_panel`（按报告期末日）保留给 Stage 1
旧因子（其 Q1 轻微 forward-looking 局限当时已在报告里披露）。

### 3. TTM EPS —— 把累计 YTD 滚成滚动 12 月

同花顺的「基本每股收益」是**累计 YTD**口径（Q1=一季度，Q2=上半年，Q3=前三季度，
Q4=全年；实测 600519 逐季递增确认）。但 PE(TTM) 需要滚动最近 12 个月的 EPS。
换算公式（`compute_ttm_eps()`）：

    TTM_EPS[本期] = EPS_YTD[本期] + EPS_YTD[上年年报] - EPS_YTD[去年同期]

年报本身即全年 = 天然 TTM。缺上年年报/去年同期的早期报告期算不出 TTM（不瞎填）。

### 4. 估值因子改写 —— `astock_quant/factors/fundamental.py`

| 因子 | T1 口径 |
|---|---|
| PE  | 当日收盘价 / 每股收益(TTM)；负 PE（亏损股）→ NaN |
| PB  | 当日收盘价 / 每股净资产(BVPS)；负 PB → NaN |
| DividendYield（新增） | 近 12 月每股现金分红 / 当日收盘价 |
| GrossMargin（新增） | 销售毛利率（质量因子） |
| ROE / NetMargin / EPS / 营收同比 / 净利同比 | 改用 publish_date 对齐 |

`default_factors()` 财务因子从 7 个扩到 9 个，总因子数 25 → 27。

经核 EPS_TTM=0.02 真实）；PB 中位数 2.7、min 0.41（银行低于净资产）；
股息率中位数 1.7%、max 9.4% —— 均符合 A股 蓝筹常识。

## 局限（如实披露）

1. **披露日是「法定截止日」近似，非每只票真实披露日**。多数公司提前披露，故本口径
   偏保守（可见时点偏晚）—— 不泄漏未来，但 4 月/8 月/10 月那几周的因子值会比
   「用真实披露日」略滞后。若日后能拿到可靠逐期披露日，可进一步精确化。
2. **同花顺财报字段口径**：EPS/BVPS/ROE 等取自同花顺源；不同数据源对同一指标口径
   可能微差（如 ROE 摊薄 vs 加权）。本实现取同花顺「净资产收益率」列，全程口径一致。
3. **分红按报告期挂载**：分红金额挂到对应报告期的财报记录、复用其 publish_date。
   分红预案实际公告日可能与年报披露日有数日~数周差异 —— 已偏保守处理（用财报
   披露日，通常 ≤ 分红除息日）。
4. **银行/保险无毛利率**：金融业无营业成本概念，毛利率字段天然 NaN（83.3% 覆盖率
   的缺口主要来自此），属正确行为，下游打分需对该因子按行业 mask。
